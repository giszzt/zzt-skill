# 五种操作模式 — 详细执行步骤

> **本文件由 skill.md 拆分而来，包含各模式的完整执行步骤。**
> 路由逻辑和通用规则见 `../skill.md`。

---

## 模式 A：代写生成

**适用输入形态：** 主题+关键词 / 详细提纲 / 零散素材片段 / 需大幅重写的草稿

> ⚠️ 若对应文体档案尚未建档（状态为🔲），告知用户并引导进入【模式 D：建库】。

**本模式为文体无关的通用流程。** 文体专属的规划要求、评审角色、通过标准，均定义在各文体档案的"代写规划扩展"章节中，由主控在执行时读取并注入。

---

### 第一阶段：规划（主控执行）

**第1步（必须）：读取 core + 文体档案**
先读取 `references/style-core.md`，完整阅读"思维倾向参考"和"稳定禁忌"。再读取对应文体档案（含其"代写规划扩展"章节）。**两个文件都读完，不得进入第2步。**

**第2步（必须）：内容规划**

按两层执行：

**第一层：通用结构规划（所有文体共用）**

- 这篇文章的顶层结论/核心信息是什么？
- 支撑该结论的2-4个核心论点是什么？
- 每个论点的支撑材料/展开路径是什么？
- 论点之间的逻辑关系是递进、并列还是因果？

**第二层：文体专属规划扩展（按需）**
若当前文体档案定义了"代写规划扩展"章节 → 读取并补充执行该章节列出的额外规划项。

**第2.5步（必须）：加载声音地基**

每次代写，无论长短，都必须执行此步。结果存入主控工作记忆，后续分别传给生成智能体和评审智能体。

**① 识别写作段落的功能类型**（当前任务主要涉及哪类段落，对照下表选 1-2 个）：

| 功能类型 | 首选样本文件 | 补充文件（可选） |
|---|---|---|
| 引言 / 背景铺垫 / 缺口建立 | `samples/引言.md` | `samples/综述收束.md` |
| 文献综述结尾 / 缺口小结 | `samples/综述收束.md` | `samples/引言.md` |
| 方法阐述（技术路线/算法） | `samples/方法-技术方案.md` | `samples/概念界定.md`（含新概念时）|
| 方法阐述（系统框架/多要素架构） | `samples/方法-系统架构.md` | `samples/逻辑关系揭示.md` |
| 案例展示 / 验证 | `samples/案例验证.md` | `samples/数据分析与实证结果.md`（含数据时）|
| 实证结果 / 数据分析 | `samples/数据分析与实证结果.md` | `samples/案例验证.md` |
| 讨论 / 展望 | `samples/讨论展望.md` | `samples/结论.md` |
| 结论 / 贡献总结 | `samples/结论.md` | `samples/论证上行.md` |
| 摘要 | `samples/摘要.md` | — |
| 核心概念定义 / 界定 | `samples/概念界定.md` | `samples/关系分析.md`（含关系论述时）|
| 两概念 / 机制之间关系 | `samples/关系分析.md` | `samples/逻辑关系揭示.md` |
| 多概念递进 / 协同关系 | `samples/逻辑关系揭示.md` | `samples/关系分析.md` |
| 论点升维 / 从局部到意义 | `samples/论证上行.md` | `samples/结论.md` |

**② 读取样本，抽取 2-3 个段落**：读选定的 1-2 个文件，挑选与当前任务**论文类型**最接近的案例（同为工程系统类 / 实践探索类等），抽取 2-3 个原文段落及其"写法要点"注释。控制在 2-3 段，不超过 4 段。

**③ 读取改写对照库**：读 `references/pattern-pairs.md`，选取与当前文体和任务类型最相关的 8-10 对。标准：学术论文侧重第一、二、五类；公众号侧重第一、二、六类；课题方案侧重第一、二、三类。第一类前三对每次必选。

以上①②③合称**声音地基**，必须完整传给生成智能体和评审智能体（评审智能体仅传②中的段落，不传③）。

### 第二阶段：生成（生成智能体执行）

**第3步（必须）：调度生成智能体**

使用 Agent 工具生成文章草稿。主控在 prompt 中传递以下信息：

> **生成智能体 prompt 要素（必须包含，按顺序排列）：**
>
> 1. **角色**：你是一位写作专家，需要按照指定的风格规则代写文章（文体由加载的风格档案决定）
> 2. **风格规则文件路径**（指示生成智能体读取）：
>    - 跨文体内核：`references/style-core.md` 的完整路径
>    - 文体档案：对应 `references/style-*.md` 的完整路径
> 3. **声音参照段落**（必须原文嵌入 prompt，不得只给文件路径）：
>    来自声音地基②的 2-3 个原文段落，附各自的"写法要点"注释。格式：
>    ```
>    【声音参照——作者本人原文段落，感受写法，不复制内容】
>    段落1（来自 samples/XX.md，功能：XX）：
>    > [原文]
>    写法要点：[注释]
>
>    段落2（来自 samples/XX.md，功能：XX）：
>    > [原文]
>    写法要点：[注释]
>    ```
>    注意事项（附在段落后）：感受节奏、选词密度、抽象-具象切换方式；不迁移领域词汇；不复制内容框架，只感受写法。
> 4. **改写对照**（必须原文嵌入 prompt，选 8-10 对）：
>    来自声音地基③的对照对，格式：
>    ```
>    【改写对照——AI 默认写法 vs 作者实际改法，生成时主动避开 AI 默认写法】
>    P01 冒号+完整句：
>      [AI]: 这意味着：系统必须……
>      [改]: 这说明系统必须……
>    P04 "不是A而是B"：
>      [AI]: 这不是技术局限，而是数据稀缺的问题。
>      [改]: 问题出在数据稀缺上。
>    [其余选取的对照对]
>    ```
> 5. **写作任务**：用户的原始需求
> 6. **内容规划**：第2步制定的完整规划内容（通用规划 + 文体专属扩展项）
> 7. **输出要求**：输出正文全文（纯文本），按章节或段落组织
> 8. **写作红线清单**（以下为逐句级硬约束，必须原文附在 prompt 中，不得省略或概括）：
>
> ```
> 【写作红线——逐句级硬约束，违反任何一条即不合格】
> ① 冒号仅用于引出正式列举或直接引语。"X的核心是：……""X意味着：……"
>   这类"冒号+完整句子"一律改为逗号衔接或重组句子，零容忍
> ② "不是A而是B"/"并非A而是B"全文不超过2处，每处须通过白名单检验：
>   读者是否真的可能持有A这个错误认知？不是则删掉"不是A"直接说"是B"
> ③ "结构性""系统性""战略性""全局性"等修饰词逐个审视，
>   去掉后语义无实质损失则删
> ④ 风格规则文件中的任何内容不得出现在正文中——包括规则本身、
>   规则的原因说明（如"受制于AI代写场景的……"）、执行约束描述。
>   正文是给读者看的，不是给写作系统看的
> ⑤ 连续两个以上子节不得遵循完全相同的叙事模式（同构检测）。
>   写第二个同类子节前，回看第一个的结构，主动变换切入角度或叙事顺序
> ⑥ "这""这一""这个""它"全文合计不超过3处，超出则改为重复核心名词或重组句子
> ```
>
> **若为修改轮次，额外附加：**
> 7. **上一轮评审意见**：评审智能体返回的完整结构化评审报告
> 8. **修改指令**：针对评审意见中标注的具体问题逐项修改，保持风格规则不变

---

### 第三阶段：评审（评审智能体执行）— 条件触发

**第4步：判断是否启用评审智能体**

**同时满足以下两个条件才启用评审，否则跳过本阶段：**

- 条件①：预计正文字数 ≥ 2000 字
- 条件②：当前文体档案的"代写规划扩展"章节中指定了评审标尺文件

> 用户可通过"跳过评审"关键词手动强制跳过，无论是否满足上述条件。

**满足条件时，使用 Agent 工具调度评审智能体，prompt 包含：**

> **评审智能体 prompt 要素（必须包含）：**
>
> 1. **评审标尺**：读取文体档案"代写规划扩展"章节中指定的评审标尺文件（包含评审角色、各维度和通过标准，含 V0 声音检查维度）
> 2. **声音参照段落**（必须原文嵌入 prompt）：
>    来自声音地基②的 2-3 个原文段落（与传给生成智能体的相同）。格式：
>    ```
>    【声音参照——作者本人原文段落，用于 V0 声音偏差检测的对照基准】
>    段落1（来自 samples/XX.md，功能：XX）：
>    > [原文]
>    段落2（来自 samples/XX.md，功能：XX）：
>    > [原文]
>    ```
> 3. **待审文章**：生成智能体返回的正文全文
> 4. **输出要求**：严格按评审标尺规定的格式输出结构化评审报告（含 V0 维度的声音偏差输出）
>
> **重要原则：** 评审智能体不加载风格规则文件（style-core.md、style-paper.md 等）。它接收的声音参照是**原文段落**（不是规则描述），用于 V0 声音对比判断，而非规则执行检查。内容质量（D1-D7）和声音偏差（V0）是两个独立检查维度，互不干扰。

---

### 第四阶段：迭代判断（主控执行）

**第5步：评估评审结果并决定是否迭代**

- **通过**：评审报告满足评审标尺中定义的"质量阈值" → 进入第五阶段
- **需修改**：未达到质量阈值 → 将评审意见发回生成智能体修改（回到第3步），修改后重新评审（重复第4步）
- **最多迭代 2 轮**。超限后输出当前最优稿 + 未解决问题清单，交由用户决策

每轮迭代时，主控应在生成智能体 prompt 中**明确引述**评审报告中未通过维度的具体意见和修改方向，不做模糊转述。

---

### 第五阶段：风格合规扫描与输出（主控执行）

**第6步（必须）：风格合规扫描**

对生成智能体（或经评审迭代后的）最终稿执行以下扫描，**逐项输出结构化扫描报告**。扫描报告是主控自己的工作记录，不输出给用户，但必须在输出正文前完成并据此修改。

> **风格合规扫描清单：**
>
> **S1 标点扫描**（对照 style-core 2.5 标点偏好表逐句执行）：
>
> - 统计全文冒号使用次数，逐个判断：是否用于引出列举或直接引语？不是则改为逗号或重组句子
> - 统计破折号、分号、括号使用次数，逐个判断是否非必要
> - 输出格式：`冒号：发现N处，修改M处（列出每处：原文片段→修改后）`
>
> **S2 禁忌句式扫描**（对照 style-core 三A 结构禁忌）：
>
> - "不是A而是B"/"并非A而是B"：统计次数，超过2处则逐个执行白名单检验并删减
> - 万能修饰词（"结构性""系统性"等）：逐个审视，无语义贡献则删
> - 显式编号过渡词（"其一/其二""首先/其次"）：发现则改为内容自然承接
> - 输出格式：`"不是…而是…"：发现N处，保留M处（保留理由），删除K处`
>
> **S3 指示代词扫描**：
>
> - 统计"这""这一""这个""它"出现次数，超过3处则改为重复核心名词或重组
>
> **S4 内部指令泄漏扫描**（对照 style-core 三D 元规则禁忌）：
>
> - 扫描正文中是否出现风格规则、评审标尺、执行约束的内容或措辞，发现则删除或改写
>
> **S5 子节同构扫描**（对照文体档案反模板检测规则）：
>
> - 对连续两个以上同级子节，提取各自的叙事结构骨架（如"痛点→方案→框架运行→关键交互点"），判断是否同构
> - 同构则对第二个及以后的子节调整切入角度或叙事顺序
>
> **扫描报告为空（声称零违规）的处理**：对于 2000 字以上的长文，S1-S5 全部零违规在统计上极不可能。若扫描报告为空，主控须对全文前三段重新逐句执行 S1 和 S2 作为抽检，确认确实无违规后才可通过。

**第6.5步（必须）：据扫描报告修改全文**

将 S1-S5 中标记的所有违规项就地修改，修改完毕后进入第7步。

**第7步（必须）：输出正文**
将最终稿输出给用户。若经过评审迭代，在正文后附简要说明：

- 评审通过的维度概况（如有）
- 仍需用户关注的问题（如有）
- **风格合规扫描摘要**（一句话，如"标点扫描修改了8处冒号、2处破折号；删除了3处'不是…而是…'句式"）

**第8步（必须）：记录 AI 稿**
输出正文后，调用观测脚本记录（`GENRE` 替换为实际文体代码）：

```bash
python scripts/observe.py record-original --text "此处粘贴生成的正文全文" --genre GENRE --mode A
```

文体代码：`paper`（学术论文）| `wechat`（公众号）| `proposal`（课题方案）| `workplan`（工作方案）
正文过长时先写临时文件：`python scripts/observe.py record-original /tmp/draft.txt --genre GENRE --mode A`

## 模式 B：润色改写

**核心目标：** 保留原文内容与逻辑，注入作者风格，消除 AI 腔与非本人惯用表达

**执行步骤（严格按顺序，不可跳步）：**

**第1步（必须）：读取 core + 文体档案**
先读取 `references/style-core.md`，再读取对应文体档案。**两个文件都读完，不得进入第2步。**

**第2步（必须）：诊断偏差**
列出原文与风格档案的主要偏差，不超过3条，简洁。

**第3步（必须）：输出润色全文**
按润色优先级处理：①语言风格统一化（消除AI腔）→ ②结构逻辑调整 → ③学术规范化 → ④压缩/扩写。

**第4步（必须）：风格合规扫描**
执行模式 A 第6步的风格合规扫描清单（S1-S5），据扫描结果就地修改全文。

**第5步（必须）：附风格改动摘要**
说明做了哪几类改动、各改了几处，含风格合规扫描的修改摘要。

**第6步（必须）：记录 AI 稿**
输出润色全文后，调用观测脚本记录本次输出：

```bash
python scripts/observe.py record-original --text "此处粘贴润色后的全文" --genre GENRE --mode B
```

## 模式 C：风格迁移

**定义：** 将一篇文章从文体 A 的风格转换为文体 B 的风格（内容框架保留，语言与结构重组）

**执行步骤（严格按顺序）：**

**第1步（必须）：读取 core + 两个文体档案**
先读 core，再读源文体档案，再读目标文体档案。**三个都读完，不得跳过。**

**第2步（必须）：输出迁移差异对照**
列出两个文体在关键风格维度上的对比，3-5条。

**第3步（必须）：执行迁移，生成目标文体版本**

**第4步（必须）：风格合规扫描**
执行模式 A 第6步的风格合规扫描清单（S1-S5），据扫描结果就地修改全文。

**第5步（必须）：附迁移说明**
哪些内容被保留、哪些被重构、哪些因文体差异被删减。

**第6步（必须）：记录 AI 稿**
输出迁移全文后，调用观测脚本记录本次输出（`GENRE` 填目标文体代码）：

```bash
python scripts/observe.py record-original --text "此处粘贴迁移后的全文" --genre GENRE --mode C
```

> ⚠️ 若任一文体档案未建档，暂停并提示用户先完成建档。

## 模式 D：建库 / 更新风格档案

**触发条件：** 用户提供新样本文章，需提炼风格指纹并写入档案

**执行步骤：**

1. 读取 `references/style-extractor.md`（风格提取提示词）
2. 读取 `references/style-core.md`（作为比较基准）
3. 对用户提供的样本执行风格提取，输出结构化风格指纹 JSON
4. **关键步骤：特征分拣**——将提取结果与 core 对比，判断每个特征属于哪一层：
   - 与 core 一致 → 标注为"core 已覆盖"，不重复写入文体档案
   - core 中有但在本文体中表现不同 → 写入文体档案的"core 在本文体中的具体表现"
   - core 中没有且为本文体独有 → 写入文体档案的"本文体特有规则"
   - core 中没有且看起来跨文体通用 → **暂存候选**，等第二个文体验证后再上提到 core
5. 若该文体**首次建档**：复制 `references/style-template.md` 创建新档案，更新本文件档案状态表
6. 若该文体**已有档案**：与现有档案对比，识别新样本带来的特征变化，更新档案并标注版本
7. **core 层迭代检查**：如果新样本来自一个新文体，检查是否有特征在两个以上文体中都稳定出现但 core 尚未收录，若有则上提到 core

**样本质量建议：**

- 单次建档投入同一文体 2-3 篇文章，识别稳定特征
- 样本字数建议 800 字以上
- 避免使用有大量引用/摘抄的文章作为样本

## 模式 E：反馈学习

**触发条件：** 用户提供其对 AI 生成内容的手动修改版本，要求对比、学习或更新规则

**适用范围：** 模式 A/B/C 的任何输出均可触发，不限于润色任务

**执行步骤（严格按顺序，不可跳步）：**

**第1步（必须）：记录用户定稿**
调用观测脚本，将用户提供的定稿写入日志，与之前的 AI 稿自动配对：

```bash
python scripts/observe.py record-final --text "此处粘贴用户的定稿全文"
```

脚本会自动匹配最近一条未配对的 AI 稿，并输出字符级修改率。若跨 session 需指定 hash：`--match <hash>`。

**第2步：定位 AI 稿**

- 若 session 中有上下文：直接使用
- 若跨 session：从上一步 record-final 的输出中找到 original_content，或要求用户提供

**第3步：提取差异**
逐段对比 AI 稿与用户定稿，将所有实质性改动按以下三类归档：

- **结构类**：段落增删、逻辑顺序调整、分点↔连贯段落切换
- **语言类**：词汇替换、句式重写、AI腔消除
- **标点类**：标点修改（仅记录与"标点偏好"规则相关的修改）

**第4步：归因分析**
对每条差异判断归属，标注以下三种之一：

- `[规则执行问题]`：现有规则已覆盖该情形，但本次未执行到位
- `[规则空白]`：现有规则未覆盖，是新出现的风格特征或偏好
- `[个人偏好-不纳入]`：属于此次特定内容的一次性调整，不具备规则化价值

**第5步（必须）：写入反馈日志**
将本次差异以结构化格式追加至 `references/feedback-log.md`（格式见该文件）。**必须写入，不得跳过。**

**第6步：频次统计与置信度评估**
读取 `references/feedback-log.md` 的完整历史记录，统计各类 `[规则空白]` 和 `[规则执行问题]` 的累计出现次数：

- **P0（≥3次）**：稳定模式，建议立即纳入或修订规则
- **P1（2次）**：重复信号，列入候选，下次验证时确认
- **P2（1次，即本次首见）**：弱信号，记录存档，暂不修改规则

**第7步：输出提案摘要**

```
【本次反馈摘要】
文体：XX ｜ 模式：X ｜ 修改条数：N ｜ 修改率：XX%

【差异一览】
| 类别 | AI稿片段 | 用户定稿 | 归因 |
|------|---------|---------|------|

【规则更新提案】
P0（建议立即应用）：
- [提案] → 目标：[style-core.md / style-XX.md] ｜ 章节：[XX]

P1（建议关注）：
- [提案] → 下次出现时确认

P2（观察存档）：
- [观察] → 首次出现，等待验证

是否应用 P0 提案？回复"应用"或"跳过"。
```

**第8步（按用户确认）：安全写入规则**

若用户确认"应用"：

1. 先调用备份脚本，保护目标文件：

   ```bash
   python scripts/improve.py backup --target core        # 若提案写入 style-core.md
   python scripts/improve.py backup --target paper       # 若提案写入 style-paper.md（按实际文体）
   ```

2. 将 P0 提案直接写入对应档案（style-core.md 或文体档案），并在其版本/迭代日志中追加记录

3. 将 P1/P2 提案追加至 `references/improve-proposals.md`，标注 `[待确认]`

4. 告知用户：如发现规则写入有误，可随时运行 `python scripts/improve.py rollback --target core` 回滚

若用户确认"跳过"：

1. 将所有提案（含 P0）追加至 `references/improve-proposals.md`，标注 `[待确认]`
2. 提示用户：积累多条后可运行 `python scripts/improve.py extract` 批量分析并生成标准提案文件
