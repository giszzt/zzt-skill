## 一、核心思维操作在课题研究方案中的具体表现

> 以下四节对应 `style-core.md` 一、思维操作习惯的四个条目。不重复定义，只说明课题方案中的特化行为。

### 1.1 缺口驱动 → 课题方案中的表现

这是课题方案中**密度最高**的思维操作，几乎每个研究内容单元都以"缺口"开头。

- 缺口出现位置：研究背景段（宏观层面）、关键技术问题各条（具体层面）、每项研究内容概述（操作层面）——三层叠加，比学术论文密集得多
- 标准句式："针对…问题，拟开展/突破/建立/研发…"
- 缺口表述偏向技术层面的具体障碍（"识别精度不足""评估标准缺乏""三维模拟能力不足"），不泛说"研究不充分"
- 转折结构多用"然而…""但…""尚未…""难以…"，随后紧跟本研究的应对策略
- 每项研究内容末尾常追加"为X提供Y支撑"，回扣解决缺口的价值链

### 1.2 关系优先 → 课题方案中的表现

- 研究内容之间的逻辑关系需明确标出，常见表达："以X为基础/支撑""X是Y的前提""Z为W提供理论/数据/技术支撑"
- 惯用链式结构呈现研究逻辑（如"识别-评估-优化""理论-技术-数据-平台-示范"），横线不是随意并列，而是揭示前后依赖关系
- 研究内容之间的支撑关系通常在每项内容末尾交代，不在开头就说

### 1.3 抽象-具象切换 → 课题方案中的表现

与学术论文相比，课题方案**重具象端**：最终要落到具体的可交付物、指标精度和应用示范，抽象层的理论阐述相对简短。

- 下行（具象化）是主方向：宏观目标 → 关键技术问题 → 具体研究子项 → 量化考核指标
- 量化指标是具象化的终点："识别精度>85%""模拟精度>70%""发表论文≥23篇"
- 抽象原则出现后，通常在下一段或同一段内就给出具体的研究内容对应项，不让抽象悬空

### 1.4 整体视角 → 课题方案中的表现

- 整体框架在"研究背景"或"研究目标"段中前置展示，后续各项研究内容在此框架下依次定位
- 对"完整链条"的偏好比学术论文更强，全流程、全链条、全维度等词汇出现频率高，用于呈现研究覆盖的完整性
- 章节末尾常有"综上"收束，汇总本项目在该维度上的整体立场

## 二、语感指纹在课题方案中的调节

> 只记录与 core 不同或需要特别强调的部分。

### 2.1 句子构建调节

- 手段-目的嵌套句式是主导句型，且高度程式化："针对X问题，研究/构建/突破Y，实现Z"是最高频的句式
- 长句为主：一个研究内容的概述段往往是一到两个长句，承载"问题-方法-目标"三层信息
- 比学术论文更多使用"以X为Y"的定语式构词（"以时空大数据为支撑""以人为核心的新型城镇化"）

### 2.2 段落构建调节

- 段落节奏不同于学术论文：概述段（一段）+ 子条目列表（⚫或①②③）是课题方案的常见段落结构，不是连贯论证段落
- 子条目列表中每项是一个研究子问题，格式压缩，不展开论证
- "框架概括句"在此文体中**通常保留**：评审读者需要从段首句预判本段处理哪个研究问题，框架句帮助定位；相比学术论文，删除的门槛更高

### 2.3 用词倾向调节

- 技术动词高频出现且偏向行动性："突破""研发""构建""建立""实现""形成""探究"——这些动词不是装饰，每一个都对应一项具体交付物
- 体系化词汇是正当用法（非禁忌）：全链条、全流程、一体化、全维度，在课题方案中用于说明覆盖范围，有实质语义，不是空洞修饰
- "多X多Y多Z"短语结构频繁使用（多目标多主体多情景、多尺度多要素多维度），用于呈现研究内容的维度覆盖，是文体惯例
- core 层"禁止万能修饰词"规则在此文体中**部分解除**：当"系统性""智能化""精准化"等词直接对应一项技术指标或功能时，保留；当用作纯强调语气时，仍删除

## 三、课题方案特有的写作规则

> 只在课题方案中适用，不属于 core 层。

### 3.1 量化成果规则

预期成果段必须量化，且按成果类型分类列举（具体分类因课题类型而异，但量化原则通用）：

- 知识产权类：论文≥N篇、发明专利≥N项、软著≥N项
- 系统/平台类：具体命名+数量
- 数据/标准类：数据库N个、标准规范N个
- 精度指标类：具体技术指标给出数字门槛（>X%）
- 人才培养类：研究生N名（基金项目常见）

不同课题类型对成果分类的要求不同，但"每类成果都有具体数字"是通用规则。

### 3.2 研究内容的三层结构

每项研究内容按以下三层展开，是跨课题类型的共同写法：

1. **问题层**（概述段开头）：针对XX现有不足/挑战，说明为什么需要开展这项研究
2. **内容层**（概述段主体）：本项目拟开展/构建/突破具体什么，用一段连贯文字交代研究路径
3. **子项层**（要点列表）：以⚫或①②③列出具体研究子问题，每项是名词短语，不展开方法细节

三层缺一不可：只有子项列表而没有概述段，读者无法理解研究逻辑；只有概述段而没有子项，研究边界不清晰。

### 3.3 关键科学问题的设问格式

关键科学问题（KSQ）的标准写法：

1. 以"如何…"或"X是如何/怎样…"开头设问（框定问题边界）
2. 背景铺垫：一句话说明为什么这是个问题（现有方法/认知的局限）
3. 解决指向：一句话说明本项目拟如何回答这个问题

设问结构比直接陈述"核心挑战是X"更有探究感，与 core 层的"声音质地"规则一致。

### 3.4 研究基础的数字化列举

可行性论证中的团队/前期基础段，采用具体成就数字支撑，不做空泛声明：

- 用序号分条，每条以具体数字或成果开头（发表高水平论文N篇、承担国家级项目N项）
- 数字指向具体成果，不说"丰富经验""深厚积累"而无数字
- 末句做整体收束，点明积累与本项目的关联，而非只列举成果

### 3.5 文献引用密度的分布规律

文献引用不均匀分布，与学术论文差异明显：

- 立项依据（研究现状综述）部分：高密度，引用贯穿全段，格式同学术论文 (Author et al., Year)
- 研究内容、研究方案、可行性分析部分：引用密度显著降低，重点是阐述本项目的方法路径，不再大量引用他人工作
- 这种分布反映了课题方案的叙事逻辑：用引用建立"现有基础+不足"，之后切换到"本项目如何做"

## 四、课题方案特有的禁忌

> 补充 core 层稳定禁忌之外的、课题方案特有的禁忌。

1. **禁止成果指标模糊化**：预期成果段中精度指标、数量指标必须给出具体数字，不能只说"显著提升""大幅提高"而无数字支撑
2. **禁止研究内容与研究目标混同**：研究目标段说"做到什么"（宏观结果），研究内容段说"做什么"（具体工作），两段不能互相复制
3. **禁止可行性悬空**：研究方案中的每项技术路线，需在研究基础段找到对应的积累支撑，不能宣称能做某事但基础段全无铺垫
4. **禁止单项研究内容孤立**：每项研究内容末尾需交代其对后续研究或整体目标的支撑作用，避免各项研究看上去互不关联

## 五、领域防火墙

样本来自城市规划、地理信息、遥感、城市更新等领域，以下词汇是领域专有术语，迁移到其他领域时需替换为对应领域术语：

- 城市信息模型（CIM）→ 替换为目标领域的"综合数据平台"类概念
- FLUS、元胞自动机（CA）→ 替换为目标领域的"模拟模型"
- 存量低效空间 → 替换为目标领域的"研究对象"
- 数字孪生 → 替换为目标领域的"仿真/虚拟化"技术概念
- 容积率（FAR）、天际线、三维紧凑度 → 替换为目标领域的定量评估指标

迁移性判断：本档案中关于**论证结构（三层结构）、段落组织（概述+子项）、成果量化格式、基础数字化列举**的规则，在任何学科的课题方案中均可直接应用；技术动词的密度在理工科课题中保留，在人文社科课题中可适度降低。

## 六、与学术论文文体的主要差异

> 帮助执行风格迁移（模式 C）时快速定位两个文体的分叉点。

| 维度 | 学术论文 | 课题方案 |
|------|---------|---------|
| 叙事重心 | 方法创新的论证过程 | 问题必要性 + 研究路径 + 可行性 |
| 缺口密度 | 集中在引言/综述 | 每个研究内容单元都有 |
| 段落结构 | 论点+论证+收束（连贯段落） | 概述段+子条目列表 |
| 成果描述 | 学术贡献（理论/方法层面） | 量化可交付物（数量+精度指标）|
| 文献引用 | 高密度，贯穿全文 | 集中在立项依据，研究内容部分明显减少 |
| 读者关系 | 同行评审（质疑性读者） | 评审专家（判断可行性+研究价值） |
| 结构灵活度 | 较灵活，可创意性组织 | 有固定章节框架，但各类型课题不同，不可套用同一模板 |
